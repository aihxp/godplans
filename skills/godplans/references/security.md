# Security and hardening planning module

Plans the Security and hardening sections of PLAN.mdx by moving secauditor controls into explicit requirements, tasks, and reproduced-attack checks before implementation. The finished project still requires runtime verification and an independent audit. The orchestrator loads this module for every archetype with any network surface, auth, persistent data, secrets, CI/CD, containers, or LLM calls; only a fully offline library with none of these may exclude it, with the reason recorded in the applicability matrix.

## Lineage

Descends from secauditor (an 11-dimension read-only vulnerability audit anchored to OWASP 2025/2021, API Top 10 2023, ASVS, NIST, SLSA, and the OWASP LLM Top 10, with source-to-sink evidence, per-dimension paper-control hunting, and a fixed automatic-Critical list that caps the overall score at 79) and harden-ready (post-deploy adversarial verification: reproduced-attack OWASP walkthroughs, control-to-code compliance mapping, class-not-instance hardening, and a functioning disclosure program). Two disciplines carry over. First, no control is planned without its wiring point and a test that proves it fires, because both ancestors hunt declared-but-unwired controls by name. Second, the automatic-Critical conditions are designed out at plan time, because any single one (user-settable role, unsigned JWT acceptance, fast-hashed passwords, live committed secret, open database ingress, prompt injection into a shell sink) invalidates every other point earned.

## Decisions to force

1. Authorization model and ownership boundary. What single mechanism decides who may touch which record, and where does it live? Hard to reverse because retrofitting ownership means rewriting every query and handler after data exists, and IDOR is the highest-weighted audit dimension (AUTHZ, 18%). Options: central deny-by-default policy middleware with ownership predicates inside the query (default); database row-level security (default for multi-tenant on Postgres or Supabase); scattered per-route checks (refused: new handlers silently bypass a blocklist). Multi-tenant designs must also pick the tenant-scoping mechanism now: RLS or a mandatory ORM filter deriving tenant id from the session, never a client-supplied header.
2. Identity architecture. Server-side sessions, self-managed JWTs, or a hosted identity provider, and which password KDF? Hard to reverse because token formats leak into every client and stored hashes cannot be migrated without a login-time upgrade path. Options: hosted OIDC provider with Authorization Code + PKCE (default when the budget allows); server-side sessions with CSPRNG ids (default for classic web apps); self-managed JWT with a strict algorithm allowlist (only with a stated reason). KDF: argon2id default; scrypt, bcrypt, or PBKDF2-HMAC-SHA256 at current cost floors acceptable; anything faster is an automatic Critical.
3. Secrets sourcing and rotation. Where do secrets live, and how do they rotate? Hard to reverse because anything ever committed is permanent in git history and must be rotated, not deleted. Options: cloud secrets manager with the SDK wiring named in the plan (default for deployed apps); platform-injected env vars (acceptable for small deployments); committed dotfiles (refused). Decide the CI credential mode now: OIDC federation over long-lived static cloud keys.
4. Data classification and encryption-at-rest scope. Which fields are sensitive or regulated, and does the deletion story reach backups, caches, indexes, and logs? Hard to reverse because retrofitting encryption and erasure onto populated stores and shipped backups is a migration project. Options: classify at schema-design time and pick the regime from the data, PCI, GDPR, HIPAA, or SOC 2 (default when regulated data exists); an explicit "no regulated data" declaration with reason (acceptable).
5. Deployment context and surface set. Internet-facing, internal, or local-only, and which conditional surfaces exist: API, uploads, outbound fetches, webhooks, containers/IaC, LLM calls? Every downstream security requirement calibrates against this; an undeclared surface becomes an unplanned control set discovered missing at audit. Default: declare every surface as present or absent in the plan, with a reason per absence.

## Plan requirements

1. R-SEC-1: The plan contains a threat-model subsection mirroring secauditor Phase 1: entry points (routes, webhooks, uploads, queue consumers, LLM inputs, CLI/env), trust boundaries with their enforcement mechanism, sensitive assets, and principals/roles with what each may and may not do, plus a STRIDE pass per important boundary.
   Criterion: WHEN the security section is emitted THE PLAN SHALL enumerate entry points, enforced trust boundaries, sensitive assets, and principals, and SHALL declare deployment context (internet-facing, internal, local) and data sensitivity.
2. R-SEC-2: The plan declares every conditional security surface (web/API, auth, DB, uploads, outbound fetches, containers/IaC, CI/CD, AI/LLM, regulated data) as present or absent so conditional control sets are planned, never discovered missing.
   Criterion: IF a surface is declared absent THE PLAN SHALL record the reason in the applicability matrix; IF present THE PLAN SHALL carry that surface's requirements onto tasks.
3. R-SEC-3: The plan specifies a central deny-by-default authorization layer every route passes through, and object-level authorization binding each load and mutation to the current user or tenant inside the query itself (findOne({id, ownerId}) pattern or RLS), explicitly covering PUT/PATCH/DELETE, list, search, and export endpoints, plus a cross-tenant isolation test for multi-tenant designs.
   Criterion: WHEN any resource endpoint is planned THE PLAN SHALL name the ownership predicate in the query and the middleware it mounts behind, and SHALL include a task whose test proves a cross-user request returns 403 or 404.
4. R-SEC-4: The plan specifies server-side role checks on every admin or privileged route with mutating verbs guarded identically to their GET siblings, bans request-settable role/isAdmin/scope/tenant fields, and mandates allowlist DTO binding for all writes (create and update) plus explicit response DTOs for all reads (no raw ORM objects, no fields='__all__').
   Criterion: WHEN a privileged route or a write endpoint is planned THE PLAN SHALL name the enforced role check and the field allowlist, and SHALL never rely on UI hiding as the control.
5. R-SEC-5: The plan states trust placement rules: JWT claims used only after signature verification, CORS never used as access control, client-supplied identity headers never trusted, signed URLs and storage keys scoped to the requester.
   Criterion: WHEN tokens, CORS, or client headers appear in the design THE PLAN SHALL state where verification happens before any claim is trusted.
6. R-SEC-6: The plan names the password KDF with per-user salts, requires constant-time comparison for every credential, token, MFA code, and webhook HMAC check, plans per-account and per-IP throttling with lockout on login, MFA-verify, and reset (routes mounted behind the limiter, not before it), breached-password screening, and recovery flows with CSPRNG single-use short-lived reset tokens, generic auth errors, and no default credentials or env==dev auth bypass branches.
   Criterion: IF the project has an auth surface THE PLAN SHALL name the KDF, the constant-time comparison function, and the rate-limit placement relative to route mounting.
7. R-SEC-7: The plan specifies the session or token design: CSPRNG session ids regenerated on privilege change, absolute and idle timeouts, server-side invalidation on logout, Secure/HttpOnly/SameSite cookies with the token never also exposed to JavaScript; for JWTs, an explicit algorithm allowlist rejecting alg:none and RS256-to-HS256 confusion with signature/exp/iss/aud validated; for OAuth/OIDC, Authorization Code + PKCE only with exact-match redirect_uri and session-bound state and nonce; MFA, if planned, enforced server-side on the actual auth path with no bypass route.
   Criterion: WHEN sessions, JWTs, or OAuth are planned THE PLAN SHALL specify each listed property, and SHALL reject Implicit and Password grants.
8. R-SEC-8: The plan mandates parameterized queries with values passed separately (never pre-substituted), allowlists for dynamic identifiers (table, column, ORDER BY), argv-array process invocation with the shell disabled and -- separators for input-derived arguments, and re-encoding at every use of stored data in queries, commands, or templates: "came from the database" confers no trust.
   Criterion: WHEN any data access or process invocation is planned THE PLAN SHALL name the parameterized API and forbid string-built queries and shell=True equivalents.
9. R-SEC-9: The plan sets auto-escaping templating as the default with a documented review gate for every raw-HTML escape hatch (|safe, dangerouslySetInnerHTML, v-html), forbids eval/exec/new Function/template compilation on user input, requires same-origin validation for redirect targets (?next=), and non-polluting merge utilities guarding __proto__.
   Criterion: IF the project renders HTML THE PLAN SHALL list the permitted escape hatches with their review gate, and SHALL forbid DOM sinks fed by location or postMessage.
10. R-SEC-10: The plan requires, for every outbound fetch of a user-influenced URL, a scheme and host destination allowlist with RFC1918/loopback/link-local/metadata-IP blocking and post-redirect resolved-IP re-checks (never a hostname blocklist); path canonicalization plus base-directory containment for every input-derived filesystem path including archive extraction; a ban on native deserialization of untrusted bytes in favor of data-only formats with schema validation; and DTDs and external entities disabled on every XML parser, not just one.
    Criterion: WHEN webhooks, import-from-URL, uploads, or parsers are planned THE PLAN SHALL name the allowlist, the canonicalization call, and the safe-parser API per surface.
11. R-SEC-11: The plan specifies crypto upfront: AEAD only (AES-GCM or ChaCha20-Poly1305), unique per-encryption nonces, CSPRNG for all security randomness, no home-rolled crypto, adequate key sizes (RSA >= 2048, EC >= P-256), keys from KMS or env with a key-id rotation scheme the code honors, TLS 1.2+ for all sensitive transit including internal DB and cache links, HSTS, and a hard rule that certificate validation is never disabled in any branch or client, including dev-reachable ones.
    Criterion: WHEN encryption, signing, or transport is planned THE PLAN SHALL name the algorithm, the key source, and the rotation scheme, and SHALL contain the never-disable-verification rule verbatim as a task acceptance condition.
12. R-SEC-12: The plan classifies sensitive data and specifies encryption at rest including backups, Cache-Control: no-store on sensitive responses, and no sensitive data in URL query strings or logs.
    Criterion: IF PII, PHI, PAN, credentials, or tokens are stored THE PLAN SHALL map each class to its at-rest protection and its logging exclusion.
13. R-SEC-13: The plan names the secrets source with SDK wiring (not just docs), gitignore entries for all credential file types before the first commit, no hardcoded fallback keys, a rotation procedure with the rule that anything ever committed is rotated rather than deleted, no client-exposed server secrets (NEXT_PUBLIC_/REACT_APP_/VITE_ prefixes), no secrets in Docker layers (--mount=type=secret), and CI secret refs never echoed, preferring OIDC federation over static keys.
    Criterion: WHEN any credential is needed THE PLAN SHALL state where it lives, how code reads it, and how it rotates, with no fallback literal anywhere.
14. R-SEC-14: The plan includes secret scanning (gitleaks, detect-secrets, or trufflehog) in pre-commit and CI, configured to fail the build, scanning full git history, with no continue-on-error, no || true, and no mass-allowlisted baseline.
    Criterion: WHEN CI is planned THE PLAN SHALL contain a task whose Verify command runs the scanner and whose acceptance forbids soft-fail flags in the workflow file.
15. R-SEC-15: The plan specifies the global security-header set (enforcing CSP without unsafe-inline or unsafe-eval, HSTS with HTTP-to-HTTPS redirect, nosniff, frame-ancestors, Referrer-Policy) mounted before the routes with a test asserting presence; a strict exact-match CORS allowlist (no reflection, no wildcard with credentials, null origin rejected); and production configuration with debug defaulting off, generic error handlers covering all exception types, no exposed Swagger, GraphQL introspection, Actuator, phpinfo, .git, or .env, no default keys, TRACE off, and secure cookie defaults not overridable later in the stack.
    Criterion: WHEN the web surface is planned THE PLAN SHALL include a header-presence test task and SHALL state the CORS allowlist mechanism as exact string match.
16. R-SEC-16: The plan sets resource controls per endpoint class: rate limits on auth, reset, OTP, search, export, and expensive endpoints; request body-size limits; server-enforced max page sizes; quotas on paid actions (SMS, email, LLM spend); and upload handling with an extension allowlist plus magic-byte validation, size caps, sanitized filenames, and storage outside the web root.
    Criterion: IF uploads or costly endpoints exist THE PLAN SHALL name the limit values or their config location per endpoint class, not a single global limiter.
17. R-SEC-17: The plan requires committed lockfiles installed with integrity-enforcing commands (npm ci, yarn --immutable, pip --require-hashes), digest-pinned base images, third-party CI actions pinned to full commit SHAs, private-registry pinning for internal package names, an ignore-scripts policy in CI and Docker, no curl|sh installs without checksums, SCA gating CI on new high or critical advisories, SBOM generated at build from the real graph, and signed artifacts pulled by immutable digest.
    Criterion: WHEN dependencies or CI are planned THE PLAN SHALL name the integrity-enforcing install command and SHALL forbid floating action tags and || true on audit steps.
18. R-SEC-18: The plan specifies CI/CD access control: least-privilege job tokens (no write-all), environment-protection gates before prod with no self-approval, enforce_admins on, force-push to protected branches blocked, and no pull_request_target or workflow_run patterns that run fork code with secrets in scope.
    Criterion: WHEN a deployment pipeline is planned THE PLAN SHALL state the token permission block and the protection rules as task acceptance conditions.
19. R-SEC-19: For API surfaces, the plan covers the API Top 10 residue: GraphQL depth, complexity, and cost limits with introspection off in prod; an API inventory tied to the OpenAPI or GraphQL schema with a version-retirement policy (no stale /v1 or /internal mounts with weaker auth); anti-automation on high-value business flows; hardened outbound clients (TLS verify on, timeouts, bounded redirects that cannot leak bearer tokens); and webhooks secured both ways: inbound receivers verify the provider HMAC with replay protection, outbound senders sign payloads and pass the SSRF allowlist.
    Criterion: IF an API surface exists THE PLAN SHALL contain each applicable control with its enforcement point, and SHALL include version retirement in the roadmap phases.
20. R-SEC-20: The plan enumerates security events to log (auth success and failure, logout, reset, MFA, token issuance, 403 denials, payments, role grants, admin CRUD, bulk export, deletion) with actor, action, target, source IP, correlation id, UTC timestamp, and outcome at a level not dropped in production; log safety at the logging layer itself (redaction in every formatter, CR/LF neutralization, append-only shipping the app's own account cannot rewrite); and alerting on auth-failure spikes and privilege escalation routed to a monitored channel.
    Criterion: WHEN logging is planned THE PLAN SHALL list the event enumeration verbatim and SHALL place redaction in the shared formatter, not per call site.
21. R-SEC-21: If regulated data is planned, the plan picks the regime from the data (PCI, GDPR, HIPAA, SOC 2) and specifies a data classification map, minimization, retention and deletion jobs whose erasure reaches backups, caches, indexes, and logs, consent enforcement in code, audited reads of regulated records, bounded third-party sharing, and DSAR code paths.
    Criterion: IF regulated data exists THE PLAN SHALL map each control to a named code path or job, never to a policy document alone.
22. R-SEC-22: If containers, K8s, or IaC are planned, the plan specifies non-root digest-pinned minimal images with HEALTHCHECK and no layer secrets; K8s securityContext defaults (runAsNonRoot not overridden per-container, allowPrivilegeEscalation false, readOnlyRootFilesystem, drop ALL capabilities), least-privilege RBAC, default-deny NetworkPolicy, and Pod Security Admission at enforce; no public buckets (public-access-block attached), no 0.0.0.0/0 to SSH, RDP, or databases, no wildcard IAM, IMDSv2 on the templates actually used; and IaC plus image scanning in CI blocking on high or critical with no soft-fail.
    Criterion: IF container or IaC files are planned THE PLAN SHALL carry these as task acceptance conditions on the files that will exist, with the scanner gate in CI.
23. R-SEC-23: If LLM features are planned, the plan specifies role-separated prompt construction treating all retrieved, fetched, and tool content as untrusted; validation of model output before any sink (never into eval, shell, SQL, HTML, or URLs unvalidated); per-action authorization plus human-in-the-loop gating every call (not just the first) for high-impact tools with scoped per-user credentials; max-iteration and cost budgets on agent loops; no secrets or authorization rules living only in the system prompt; ACL and tenant filters inside the vector similarity query itself; pinned hash-verified model sources; and per-user rate, token, and spend limits.
    Criterion: IF the project calls an LLM THE PLAN SHALL trace each model input to its trust label and each model output to its validated sink, and SHALL name the loop budget values.
24. R-SEC-24: The plan adopts the anti-paper-control rule: every security control is specified with where it wires onto the execution path and a test proving it fires (a request that should 403, a header-presence assertion, a scanner that fails the build); and the plan designs out every automatic-Critical condition: no user-settable role fields, no unsigned-JWT acceptance, no fast-hashed or reversible passwords, no committed live secrets, no public buckets or open database ingress, no prompt-injection path into shell, eval, or SQL.
    Criterion: WHEN any security control appears in the plan THE PLAN SHALL pair it with its mount point and its firing test; IF a design choice would create an automatic-Critical condition THE PLAN SHALL be revised before emission.
25. R-SEC-25: The plan's final verification phase includes hardening checks in the harden-ready mold: reproduced-attack verification (a real cross-tenant request, a JWT alg:none rejection check, an RLS check via the anon-key endpoint where applicable), a disclosure baseline (SECURITY.md plus /.well-known/security.txt with a named inbox owner for deployed apps), class-level regression guards for any finding class fixed during the build, and a continuous cadence with a next execution date.
    Criterion: WHEN the Verification phase is emitted THE PLAN SHALL name the exact commands or requests that reproduce each check, and SHALL not accept scanner output alone as the posture.

## Self-audit rubric

- Threat model and surface declaration (10): entry points, enforced trust boundaries, assets, principals, STRIDE notes, deployment context, and every conditional surface declared present or absent with reasons.
- Authorization and tenancy (15): central deny-by-default layer named, ownership predicates in queries covering writes, list, search, export; role checks on privileged routes; no user-settable privilege fields; isolation test planned.
- Authentication and sessions (12): KDF named, constant-time comparisons, brute-force defenses behind correctly-ordered mounts, session or JWT properties specified, OAuth PKCE-only, recovery flows without enumeration.
- Injection and input handling (13): parameterized queries, identifier allowlists, argv exec, escape-hatch review gate, SSRF destination allowlist, path canonicalization, deserialization ban, XXE off, second-order re-encoding.
- Crypto and data protection (10): AEAD, nonce discipline, CSPRNG, key sourcing with rotation, TLS everywhere, cert validation never disabled, data classified with at-rest and no-store rules.
- Secrets management (8): source and SDK wiring named, rotation-not-deletion rule, no client-exposed or layered secrets, scanning that fails the build over full history.
- Misconfiguration and resource controls (8): header set with presence test, exact-match CORS, production hardening list, per-endpoint-class rate limits and caps, upload handling.
- Supply chain and CI/CD (8): integrity installs, SHA-pinned actions, digest-pinned images, SCA gate without soft-fail, SBOM, least-privilege tokens, environment gates, no PPE patterns.
- Logging, privacy, and API residue (6): event enumeration, formatter-level redaction, alerting to a monitored channel; regulated-data controls mapped to code paths; API Top 10 residue covered where applicable.
- Conditional surfaces (4): IaC and LLM requirements carried onto tasks when present, or excluded with a stated reason; never silently omitted.
- Anti-paper-control and verification handoff (6): every control paired with mount point and firing test; automatic-Critical conditions designed out; final phase reproduces attacks with exact commands, disclosure baseline and cadence dated.

Total: 100. Any plan scoring below 85 on this rubric gets revised before emission.

## Anti-patterns refused

- Paper controls (secauditor): a validator, policy decorator, or redact() defined but never on the execution path. Refusal: no control enters the plan without its mount point and a test proving it fires.
- Scanner-first security (harden-ready): a green Snyk or CodeQL run presented as the posture while an incrementing object ID leaks every tenant. Refusal: the verification phase budgets manual reproduced-attack checks with exact requests.
- Paper trust boundary (harden-ready, the Lovable CVE-2025-48757 class): a boundary declared in the architecture but unenforced in code. Refusal: every trust boundary in the threat model names its enforcement mechanism at plan time.
- Compliance-without-security (harden-ready): the checklist is green and the app is exploitable. Refusal: every compliance control maps to a named code path, config, or job, plus the question "what does the adversary do if this is weakly implemented".
- CVE-of-the-week (harden-ready): patching the instance and leaving the class. Refusal: any finding class fixed during the build gets a class survey and a class-level regression guard planned as a task.
- Hardening-as-ritual (harden-ready): an annual pen test with nothing between. Refusal: the plan declares a continuous cadence with a next execution date.
- Shallow-audit trap (harden-ready): findings a third party cannot reproduce. Refusal: every security acceptance condition is grep-verifiable or request-reproducible with the exact command in the Verify line.
- Vague-recommendation ban (secauditor): "validate input", "harden the config", "improve security". Refusal: every requirement names the safe pattern, the file it lands in, and the command that confirms it.
- Silent surface exclusion (secauditor conditional dimensions): skipping uploads, IaC, or LLM controls because nobody declared the surface. Refusal: the applicability matrix records every surface as present or absent with a reason before any section is written.
- Automatic-Critical blindness (secauditor): shipping a design where one condition caps the audit at 79. Refusal: the R-SEC-24 design-out list is checked against the draft plan before emission, and any hit forces a revision.
