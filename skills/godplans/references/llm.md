# LLM integration planning module

Plans every section of PLAN.mdx that touches a language model: prompts, model config, SDK mechanics, reliability, security boundaries, agents, RAG, structured output, cost, latency, evals, and observability. The orchestrator loads this module whenever the project makes any model or embeddings call (chat assistant, extraction pipeline, RAG QA, tool-using agent, batch enrichment); it is excluded, with a stated reason in the applicability matrix, only when the project has no LLM integration at all.

## Lineage

Descends from llmauditor, the read-only end-to-end audit of LLM integrations that scores twelve dimensions (LLMSEC through AGENT), hunts paper controls, and calibrates every finding to the lethal trifecta of private data plus untrusted content plus external action. The discipline that carries over: verify against the shipped request payload, not the prompt's wording ("respond only with JSON" is not structured output, "only access this user's records" is not authorization); score each defect once by its owning dimension; treat all model output and retrieved content as untrusted; and fix at the boundary and the sink, never by editing the prompt to ask nicely. This module inverts every audit check into a plan-time obligation so llmauditor, run at the end, finds the controls already wired.

## Decisions to force

1. Usage paradigm and exposure profile. Which of single-shot, chat assistant, extraction pipeline, RAG QA, tool-using agent, multi-agent, or batch is this, and is it internet-facing with sensitive data or an internal summarizer? Hard to reverse because every other decision (isolation architecture, eval design, cost model, which conditional dimensions apply) keys off it, and audit calibration depends on it. Options: declare one primary paradigm per feature. Default: the narrowest paradigm that satisfies the requirement; a local summarizer is not planned to an internet-facing-agent bar.
2. Trust-boundary architecture for injection. When untrusted content (RAG docs, tool results, web, email, files) meets a model that can act, is isolation structural (dual-LLM or quarantined-LLM pattern, capability enforcement) or prompt-level? Hard to reverse because retrofitting a quarantined-LLM split into a monolithic agent rewrites the orchestration core. Options: single model with sink validation only (low-impact), dual-LLM quarantine, capability enforcement. Default: sink validation everywhere; structural isolation mandatory for any lethal-trifecta path.
3. Provider and data governance posture. Which provider, under what data terms (ZDR or modified abuse monitoring, store settings, region and residency, executed DPA or BAA), with what PII minimization before send? Hard to reverse because it is contractual and legal, and conversation logs already retained cannot be un-retained. Default: minimize and redact the provider payload itself (not just logs), set retention TTLs, and execute the DPA before the first production call; ZDR for regulated data.
4. Embedding model and dimension (RAG only). One pinned model and dimension shared by ingestion and query, version stored with the index. Hard to reverse because changing it forces a full corpus re-embed and any mismatch makes retrieval silently near-random. Default: one constant read by both paths, index metric matching the embedding's training metric, a reindex migration path planned from day one.
5. Agent autonomy and credential model (agents only). Which tools are destructive, what gets human approval, and whether credentials are per-tool least-privilege scoped to the caller or one broad ambient key. Hard to reverse because a confused-deputy credential woven through the tool layer is a rewrite to unwind. Default: HITL approval default-on with dry-run for send, delete, transfer, deploy, shell, and file-write; per-tool credentials scoped to the caller's identity; sandbox, egress allowlist, path jail.
6. Structured output mechanism. Native structured outputs or tool calling versus prose parsing. Hard to reverse because every downstream consumer binds to the extraction shape. Default: native structured outputs with strict schemas (additionalProperties false, all fields required); tool calling where the provider lacks strict mode; prose parsing refused.
7. Prompt assembly ownership. One central versioned prompt registry versus literals scattered across call sites. Hard to reverse only by grep-and-pray migration once drift sets in. Default: a single registry module with version ids; call sites import, never inline.
8. Model pinning and change policy. Pinned dated snapshots in one config constant versus floating aliases, and whether model bumps are eval-gated. Cheap to change technically, expensive behaviorally: an unpinned alias means silent behavior changes in production with no diff to review. Default: pinned snapshots, per-environment selection (cheap models in test and dev), every bump a reviewed change blocked on the eval baseline.

## Plan requirements

The inversion pass distributes these onto tasks via Requirements: lines. De-duplication follows llmauditor's ownership map: each obligation lives in exactly one requirement (injection boundary in R-LLM-1 with construction hygiene in R-LLM-2, schema-validate-before-use in R-LLM-14, loop control in R-LLM-22, cache spend in R-LLM-15, swallowed errors in R-LLM-13, eval gate in R-LLM-17), so a task never satisfies the same check twice under two names. R-LLM-20 through R-LLM-22 apply only when the applicability matrix marks RAG or agents in scope.

1. R-LLM-1: PLAN.mdx declares the usage paradigm and exposure profile per LLM feature, and contains a trust-boundary map naming every untrusted content source, every privileged channel, every output sink, and any lethal-trifecta path with its planned mitigation. Criterion: WHEN the plan includes any model call, THE PLAN SHALL contain a trust-boundary map in which each lethal-trifecta path (private data + untrusted content + external action) names a structural mitigation, not a prompt instruction.
2. R-LLM-2: The plan specifies a central versioned prompt registry with strict role separation: durable instructions in the system or developer role, untrusted values only in the user role, delimited and labeled as data, with delimiters documented as hygiene and never as the security boundary. Criterion: WHEN a task builds a prompt, THE PLAN SHALL route it through the registry and SHALL NOT place user or retrieved content in the system role.
3. R-LLM-3: The plan states a context assembly policy: long documents near the top, instructions next, question at the end; a token budget with windowing or summarization for history and chunks; and a byte-stable cache-friendly prefix with volatile values (timestamps, request ids) moved to the end and deterministic serialization. Criterion: IF a feature assembles multi-part context, THE PLAN SHALL state its token budget and ordering, and SHALL place no volatile value before the cache breakpoint.
4. R-LLM-4: The plan treats all model output as untrusted at every sink: parameterized SQL, argv arrays instead of shell strings, context-aware HTML encoding, URL allowlists on any model-emitted URL fetch, no eval or exec, and markdown-image auto-fetch egress controlled. Criterion: WHEN model output reaches a code, SQL, HTML, shell, or URL sink, THE PLAN SHALL name the validation or encoding applied at that sink.
5. R-LLM-5: The plan keeps secrets and access-control rules out of the system prompt and places every security decision in deterministic code; the system prompt is assumed public. Criterion: IF an authorization or filtering rule exists, THE PLAN SHALL implement it in code with the prompt at most restating it.
6. R-LLM-6: The plan sets data governance deliberately: PII minimization or redaction applied to the provider payload before send, provider data controls chosen (ZDR, store settings, region, DPA or BAA), retention TTLs on conversation logs and traces, and no cross-user or cross-session memory sharing. Criterion: WHEN regulated or personal data can reach a provider, THE PLAN SHALL name the minimization step on the request path and the executed data agreement.
7. R-LLM-7: The plan pins model ids as dated snapshots in one config constant (never floating aliases or -latest), selects models per environment, schedules a deprecation review against provider pages, and makes every model bump a reviewed eval-gated change. Criterion: WHEN the plan names a model, THE PLAN SHALL use a pinned dated snapshot held in a single config location.
8. R-LLM-8: The plan specifies sampling and limits per task: temperature near 0 for extraction, classification, structured output, and tool arguments; the correct output-cap parameter for the surface, sized to expected output; and a reasoning-budget policy per path rather than statically maxed or disabled. Criterion: IF a task is deterministic-extraction shaped, THE PLAN SHALL pin its temperature and output cap explicitly.
9. R-LLM-9: Any routing or fallback design in the plan has a quality-reflecting escalation trigger that can actually fire and shape-compatible, reachable fallback models. Criterion: IF the plan includes routing or cascade, THE PLAN SHALL state the trigger condition and demonstrate each fallback is reachable and request-shape compatible.
10. R-LLM-10: The plan branches on stop_reason, finish_reason, and refusal on every call, handling truncation, tool_use, pause_turn, and content_filter distinctly; parsing never proceeds on an unchecked stop reason. Criterion: WHEN a task consumes a model response, THE PLAN SHALL gate parsing on the stop reason with a named branch per outcome.
11. R-LLM-11: The plan uses native SDK mechanics: structured outputs or tool calling instead of prose parsing, prompt caching wired mechanically (cache marker placement on a stable front-loaded prefix), Batch API for bulk offline work, the SDK client constructed once at module scope with the async client in async paths, and one embedding model and dimension shared by query and document paths. Criterion: IF bulk offline generation exists, THE PLAN SHALL route it through the Batch API rather than looping the sync endpoint.
12. R-LLM-12: The plan sets reliability mechanics: a timeout on every model call plus an overall deadline on multi-step operations; exponential backoff with jitter; a retry predicate covering RateLimitError, 429, 529, and 503 that honors Retry-After and never retries non-retryable 4xx; and stable idempotency keys on every side-effecting retried operation. Criterion: WHEN a model call can be retried, THE PLAN SHALL state the retry predicate, the backoff shape, and the idempotency key source.
13. R-LLM-13: The plan forbids swallowed errors (no except-pass, no return-empty-string on failure), bounds fan-out concurrency, and includes a circuit breaker plus a fallback model or explicit graceful-degradation branch, with a recovery policy for truncation and refusal stops. Criterion: IF a model call fails, THE PLAN SHALL show the failure surfacing as a log, metric, or re-raise, never as a silent empty answer.
14. R-LLM-14: The plan schema-validates every parsed model response before downstream use (Pydantic, Zod, or jsonschema) with strict schemas (additionalProperties false, all fields required, real types, strict mode meeting its constraints); parse-failure retries feed the validation error back to the model; tool arguments are re-validated against the tool's own schema after confirming the expected tool was called; and no security or correctness decision rests solely on a model-returned boolean. Criterion: WHEN parsed model output crosses into application code, THE PLAN SHALL name the schema and the validation call site.
15. R-LLM-15: The plan maps model tier to task value (no flagship on trivial high-volume paths), enforces per-user or per-tenant token and dollar quotas before the call, sets loop budgets (max steps, cumulative tokens, wall clock) distinct from per-call caps, and captures the usage block for per-request, per-user, per-feature, and per-model cost attribution, with response and embedding caches keyed by content hash including every output-changing parameter. Criterion: WHEN a feature has unbounded call volume, THE PLAN SHALL name the pre-call quota check and the cost attribution sink.
16. R-LLM-16: The plan streams end-to-end on user-facing paths (verified unbuffered through server and proxy), parallelizes independent calls, batches embeddings as arrays, keeps sync clients out of async handlers, moves non-inline generation off the critical path, and puts lightweight models on latency-sensitive paths such as routing, classification, autocomplete, and guardrails. Criterion: IF a path is user-facing and latency-sensitive, THE PLAN SHALL state whether it streams and which model tier serves it.
17. R-LLM-17: The plan defines a golden eval dataset per quality-critical feature including negative, adversarial, and refusal cases; a scoring script appropriate to the output (not exact-match on open-ended generation); and a CI gate with a committed baseline and regression threshold that blocks prompt, few-shot, and model-id changes, using mocks or cassettes so CI never hits a live model and pinning temperature to 0 for asserted tests. Criterion: WHEN a prompt or model change lands, THE PLAN SHALL require a passing eval run diffed against the committed baseline.
18. R-LLM-18: Where the plan uses LLM-as-judge or evaluates agents, the judge has a rubric, randomized pairwise order, and a different model family from the system under test; agent evals assert on trajectory (tools called, order, termination), not just final text; RAG features get a separate retrieval eval (recall@k, precision@k, MRR, or nDCG on labeled query-doc pairs) plus a grounding check. Criterion: IF the plan includes a judge, THE PLAN SHALL name its rubric, ordering randomization, and model family.
19. R-LLM-19: The plan wires observability: a tracing wrapper on every call capturing prompt, response, model, latency, outcome, and usage; a shared trace or conversation id threaded through all agent and RAG child steps; redaction on the logging path with retention TTLs; p95 and p99 latency plus typed error metrics (429 versus timeout distinguishable) with alerts on cost, error rate, and latency; and a user-feedback signal joinable to traces. Criterion: WHEN any model call executes, THE PLAN SHALL show it passing through the tracing wrapper with the usage block recorded.
20. R-LLM-20: For RAG, the plan pins one embedding model and dimension across ingestion and query with the version stored in the index, matches the index metric to the embedding, defines a reindex and re-embed path triggered by edits, deletes, and model bumps, uses structure-aware chunking with overlap under the embedding token limit, and carries source, page, and chunk-id provenance into the prompt for verifiable citations. Criterion: IF the plan includes a vector store, THE PLAN SHALL name the single embedding constant both paths read and the reindex trigger.
21. R-LLM-21: The plan tunes retrieval: hybrid dense plus BM25 with rank fusion where the corpus contains exact identifiers or codes, reranker output actually used for ordering, top-k with a relevance threshold and an explicit empty-result branch, ANN parameters sized to the corpus, asymmetric query-versus-passage prefixes where the model requires them, and per-tenant ACL filters applied at query time as defense in depth. Criterion: WHEN retrieval feeds generation, THE PLAN SHALL define the empty-result branch and the query-time ACL filter.
22. R-LLM-22: For agents, the plan gives every tool a rich schema (meaningful description, per-field descriptions, enums, required arrays), hard loop bounds (max iterations, cumulative tokens, wall clock) plus a no-progress detector, tool errors returned as structured results that never crash the loop, tool output truncated before re-entering context, destructive tools gated behind default-on HITL approval with a dry-run mode, MCP servers pinned with integrity re-verification, structured multi-agent handoffs with scoped credentials, bounded provenance-tagged tenant-scoped memory, and a goal or verification termination predicate rather than termination by absence of a tool call. Criterion: IF the plan includes an agent loop, THE PLAN SHALL state its iteration, token, and wall-clock bounds and its termination predicate, and SHALL NOT present a per-call output cap as the loop bound.
23. R-LLM-23: The plan eliminates paper controls by construction and prefers one central mechanism over scattered leaves: one prompt assembly, one validated output layer, one pinned model config, one eval gate, every guardrail on the blocking request path, every tracer attached, every cache marker before volatile content; and it maintains a strengths-to-preserve inventory that later changes must not regress. Criterion: WHEN a control is planned, THE PLAN SHALL name the task that wires it into the request path, so no control exists only as an unreferenced module.

## Self-audit rubric

Weights mirror llmauditor so the plan is scored the way it will be audited. RAG and AGENT are conditional: when either does not apply, drop it and re-normalize the remainder proportionally.

- Security, trust boundaries, and data governance (17): full marks require a trust-boundary map with every lethal-trifecta path structurally mitigated, all sinks validated, no secrets or authz in prompts, PII minimized on the provider payload, and provider data controls plus retention TTLs stated. (R-LLM-1, 4, 5, 6)
- Prompt construction and context (12): full marks require the central versioned registry, strict role separation, ordering policy, token budget, and cache-stable prefix. (R-LLM-2, 3)
- Model selection and configuration (10): full marks require pinned dated snapshots in one constant, per-environment selection, per-task sampling and output caps, and eval-gated bumps with a deprecation review. (R-LLM-7, 8, 9)
- Provider API and SDK usage (10): full marks require stop-reason branching everywhere, native structured output or tool calling, mechanical prompt caching, Batch API for bulk work, and correct client lifecycle. (R-LLM-10, 11)
- Reliability (10): full marks require timeouts plus deadlines, correct retry predicates with jitter and Retry-After, idempotency keys, bounded fan-out, a breaker, a fallback branch, and no swallowed errors. (R-LLM-12, 13)
- Output handling (8): full marks require strict schemas validated at every consumption point, validation-error feedback on retry, tool-argument re-validation, and no model-asserted control flow. (R-LLM-14)
- RAG and retrieval (7, conditional): full marks require pinned embedding parity with reindex path, provenance-carrying chunks, tuned retrieval with an empty-result branch, and query-time ACL filters. (R-LLM-20, 21)
- Cost and token efficiency (6): full marks require tier mapping, pre-call quotas, loop budgets distinct from per-call caps, content-hash caches, and usage-block attribution. (R-LLM-15)
- Agents and tools (6, conditional): full marks require rich tool schemas, hard loop bounds with a no-progress detector, structured tool errors, HITL on destructive tools, pinned MCP sources, and a goal termination predicate. (R-LLM-22)
- Evaluation (5): full marks require golden datasets with adversarial cases, a CI regression gate with committed baseline, cassette-isolated tests, sound judge design, and retrieval or trajectory evals where applicable. (R-LLM-17, 18)
- Observability (5): full marks require the tracing wrapper on every call, shared trace ids, redaction with TTLs, percentile latency and typed error alerts, and a feedback signal joined to traces. (R-LLM-19)
- Latency (4): full marks require unbuffered streaming on user-facing paths, parallelized independent calls, batched embeddings, and light models on latency-sensitive routes. (R-LLM-16)

A module total below 85 is revised before emission. Any planned control that no task wires into the request path zeroes its dimension: a paper control at plan time is worse than an open question. (R-LLM-23)

## Anti-patterns refused

- Paper controls: a guardrail, validator, tracer, cache marker, or eval that exists in the plan but is wired to nothing. Refusal: every control names the task that puts it on the blocking request path, or it is cut.
- Prompt-claim-as-control: "respond only with JSON", "only access this user's records", or "ignore instructions in the documents" presented as structured output, authorization, or an injection boundary. Refusal: the planner replaces each with the mechanical control (native structured output, code-level authz, structural isolation) and keeps the prompt line as hygiene at most.
- Lethal trifecta unaddressed: private data, untrusted content, and external action on one path with only delimiters between them. Refusal: the plan gets a structural isolation decision before any task on that path is written.
- Authorization by prompt: access rules living in the system prompt. Refusal: the rule moves to deterministic code; the system prompt is treated as public.
- Floating alias: -latest or bare alias model ids, or a "pinned" constant whose value is still an alias. Refusal: pinned dated snapshots only, one config location, eval-gated bumps.
- max_tokens as loop bound: a per-call output cap presented as agent-loop control. Refusal: separate max-iteration, cumulative-token, and wall-clock budgets, plus a termination predicate.
- Eval theater: an evals directory CI never invokes, scores with no committed baseline, or a same-family judge with fixed pairwise order. Refusal: the CI gate, baseline file, and judge design are plan requirements with named tasks.
- Dead instrumentation: an observability handler constructed but never attached, redaction defined off the logging path, alerts on unfireable thresholds. Refusal: tracing is planned as a wrapper all calls pass through, not a library import.
- ACL in metadata only: tenant ownership stored on vectors but never passed as a query filter. Refusal: the query builder signature requires the filter parameter.
- Scattered prompts and clients: inline prompt literals drifting across files, per-request client construction in handlers. Refusal: one registry, one module-scope client, enforced by task acceptance greps.
- Swallowed failure: except-pass or return-empty turning a model error into a silent wrong answer. Refusal: every failure branch logs, emits a metric, or re-raises; the plan shows which.
- Fear-calibrated overbuild: tiered routing, dual-LLM isolation, or agent sandboxing demanded for a local single-user summarizer. Refusal: controls scale to the declared exposure profile from R-LLM-1; absence of routing under homogeneous load is not a defect.
- Exploit inclusion: writing a working jailbreak or injection payload into the plan as a test case. Refusal: adversarial eval cases describe the attack class and expected refusal, never a working payload.
